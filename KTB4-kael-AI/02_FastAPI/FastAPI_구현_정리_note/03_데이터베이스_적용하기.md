## 1. 왜 DB가 필요한가

**code_vanila의 한계**

```python
_users = []  # 서버를 재시작하면 사라짐
```

서버를 껐다 켜면 모든 데이터가 초기화된다. 실제 서비스에서는 데이터를 영구적으로 저장해야 한다.

| 저장 방식 | 서버 재시작 후 | 예시 |
|-----------|--------------|------|
| 메모리 (리스트) | 데이터 사라짐 | code_vanila |
| 파일 (DB) | 데이터 유지 | code_DB |

SQLite는 파일 하나(`community.db`)로 동작하는 가장 단순한 DB다.

## 2. ORM이란

**ORM (Object Relational Mapping)** — SQL을 직접 작성하지 않고, 파이썬 클래스로 DB를 다루는 방법

```
SQL 직접 작성                   ORM (SQLModel)
─────────────────────           ─────────────────────────────
SELECT * FROM user              session.exec(select(User)).all()
WHERE email = 'a@b.com'         .where(User.email == 'a@b.com')

INSERT INTO user                user = User(email=..., nickname=...)
VALUES (...)                    session.add(user)
                                session.commit()
```

SQL을 몰라도 파이썬 문법으로 DB를 다룰 수 있다. 우리는 **SQLModel**을 사용한다.

## 3. SQLModel 기초

SQLModel은 FastAPI 개발자가 만든 ORM 라이브러리다.

```bash
pip install sqlmodel
```

### 테이블 정의

```python
from typing import Optional
from sqlmodel import Field, SQLModel

class User(SQLModel, table=True):       # table=True → 실제 DB 테이블로 만들겠다
    id: Optional[int] = Field(default=None, primary_key=True)  # 자동 증가 ID
    email: str = Field(unique=True)     # 중복 불가
    password: str
    nickname: str
    profile_image: Optional[str] = None  # 없어도 되는 필드
```

- `table=True` — 이 클래스가 DB 테이블임을 선언
- `primary_key=True` — 이 필드가 고유 식별자 (DB가 자동으로 번호를 매겨줌)
- `Field(unique=True)` — 같은 값이 두 번 들어올 수 없음
- `Optional[str] = None` — 없어도 되는 필드

## 4. database.py — DB 연결 설정

```python
from sqlmodel import SQLModel, create_engine, Session

DATABASE_URL = "sqlite:///./community.db"   # 파일 경로
engine = create_engine(DATABASE_URL)         # DB 연결 객체


def create_db_and_tables():
    SQLModel.metadata.create_all(engine)     # 모든 테이블 생성


def get_session():
    with Session(engine) as session:
        yield session
```

- `engine` — DB 파일에 연결하는 객체. 앱이 시작될 때 한 번만 만든다.
- `Session` — DB에 실제로 쿼리를 보내는 객체. 요청마다 새로 만들고 닫는다.
- `create_db_and_tables()` — 정의된 테이블이 없으면 만들어 준다. 이미 있으면 건드리지 않는다.

### main.py에서 서버 시작 시 자동 실행

```python
@app.on_event("startup")
def on_startup():
    create_db_and_tables()
```

서버가 켜질 때 `community.db` 파일과 테이블을 자동으로 생성한다.

## 5. 모델 정의하기

### 우리 프로젝트의 테이블 4개

```python
class User(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    email: str = Field(unique=True)
    password: str
    nickname: str
    profile_image: Optional[str] = None

class Post(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    title: str
    content: str
    image: Optional[str] = None
    views: int = 0
    user_id: int               # 작성자 ID (User 테이블의 id)

class Comment(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    content: str
    post_id: int               # 어느 게시글의 댓글인지
    user_id: int               # 누가 썼는지

class Like(SQLModel, table=True):
    id: Optional[int] = Field(default=None, primary_key=True)
    post_id: int
    user_id: int
```

## 6. DB에서 데이터 읽고 쓰기

### 데이터 추가 (INSERT)

```python
with Session(engine) as session:
    user = User(email="a@b.com", nickname="테스터", password="1234")
    session.add(user)      # 추가 예정으로 등록
    session.commit()       # 실제로 DB에 저장
    session.refresh(user)  # DB에서 다시 읽어옴 (id가 채워짐)
    print(user.id)         # → 1
```

`commit()` 전까지는 DB에 저장되지 않는다. `refresh()`를 해야 DB가 자동으로 채워준 `id` 값을 가져올 수 있다.

### 데이터 조회 (SELECT)

```python
from sqlmodel import select

with Session(engine) as session:
    # 전체 조회
    users = session.exec(select(User)).all()

    # 조건 조회
    user = session.exec(select(User).where(User.email == "a@b.com")).first()

    # id로 직접 조회
    user = session.get(User, 1)
```

### 데이터 수정 (UPDATE)

```python
with Session(engine) as session:
    post = session.get(Post, post_id)
    post.views = post.views + 1   # 필드 수정
    session.add(post)
    session.commit()
```

### 데이터 삭제 (DELETE)

```python
with Session(engine) as session:
    post = session.get(Post, post_id)
    session.delete(post)
    session.commit()
```

## 7. 우리 코드에서 어떻게 쓰이는가

### code_vanila vs code_DB — 모델만 바뀐다

```python
# code_vanila/models/user_model.py
_users = []

def add_user(user: dict):
    _users.append(user)
    return user

# code_DB/models/user_model.py
def add_user(user: dict) -> dict:
    with Session(engine) as session:
        db_user = User(**user)   # dict를 User 객체로 변환
        session.add(db_user)
        session.commit()
        session.refresh(db_user)
        return db_user.model_dump()  # 다시 dict로 변환해서 반환
```

함수 이름과 반환 형식이 같아서 **컨트롤러와 라우터 코드는 변경 없이** 그대로 동작한다.

### model_dump() — SQLModel 객체 → dict 변환

컨트롤러는 `dict`를 기준으로 동작하기 때문에, 모델 함수에서 SQLModel 객체를 `model_dump()`로 변환해서 반환한다.

```python
user = session.get(User, 1)
print(user)             # User(id=1, email='a@b.com', ...)
print(user.model_dump()) # {'id': 1, 'email': 'a@b.com', ...}
```

### ID 자동 증가

code_vanila에서는 ID를 수동으로 계산했다.

```python
# code_vanila — 위험: 데이터 삭제 시 ID 충돌 가능
"id": len(get_all_users()) + 1
```

code_DB에서는 DB가 자동으로 증가시켜 준다. `id` 필드를 dict에서 빼고, `commit()` 후 `refresh()`로 받아오면 된다.
